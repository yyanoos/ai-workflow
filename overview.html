AI Workflow

AI가 만들고, 테스트가 지킨다. 두 가지 워크플로우로 동작한다.

/gen-api-tests — 테스트 커버리지 구축 /dev — TDD 기반 개발

/gen-api-tests — 테스트 커버리지 구축

기존 프로젝트에 테스트가 없을 때, 엔드포인트 1개씩 통합 테스트를 만들어간다. 단일 세션 권장.

1 한눈에 보는 흐름

Phase 0
프로젝트 분석 project-analyzer 1회

PROJECT_CONTEXT.md + [AI-CONTEXT] 주석 + 테스트 인프라(docker-compose.test.yml, 프로필, 의존성, DB 스키마)

멈춤
Phase 1
엔드포인트 스캔 endpoint-scanner 1회

모든 API 엔드포인트를 찾아 endpoints.json 생성 (git 커밋)

멈춤

이하 엔드포인트 1개씩 반복

Phase 2
테스트 생성 + 실행 test-writer 반복

소스 분석 → 테스트 작성 → 실행 → 실패 시 자동 수정 (최대 3회)

첫 엔드포인트: 베이스 클래스 + support/ 헬퍼도 생성

 
Phase 3
AI 리뷰 test-reviewer

테스트 의도 적합성 검증. NEEDS_WORK 시: 반영 → 재실행 → 2차 리뷰 (1회 고정)

2차에도 NEEDS_WORK: 남은 지적사항을 보고에 포함, 사람이 판단

멈춤
Phase 4
승인 사람

검수 후 재실행 → done 처리 → 다음 pending 엔드포인트의 Phase 2 자동 시작

 

2 상태 머신

pending → Phase 2+3 → review → 사람 승인 → done
Phase 2 실패 → failed → --retry-failed → pending
pending

테스트 미생성

review

통과 + AI 리뷰 완료, 사람 검수 대기

done

사람 승인 완료

failed

3회 실패, 수동 개입 필요

3 에이전트 구성

project-analyzer Phase 0

시니어 아키텍트. 프로젝트 전체를 분석하여 AI가 이해할 수 있는 컨텍스트를 만든다.

산출물: PROJECT_CONTEXT.md, [AI-CONTEXT] 주석, 테스트 인프라

Tools: Read, Write, Edit, Grep, Glob, Bash

endpoint-scanner Phase 1

시니어 백엔드 엔지니어. 모든 API 엔드포인트를 찾아 JSON으로 정리한다.

산출물: endpoints.json

Tools: Read, Grep, Glob, Bash

test-writer Phase 2

시니어 테스트 엔지니어. 테스트 작성 + 실행 + 실패 수정까지 책임진다.

산출물: 통합 테스트 파일, support 헬퍼

Tools: Read, Write, Edit, Grep, Glob, Bash

test-reviewer Phase 3

시니어 QA 엔지니어. 통과한 테스트의 의도 적합성을 검증한다. 읽기 전용.

산출물: PASS / NEEDS_WORK 판정

Tools: Read, Grep, Glob

메인 세션 (오케스트레이터)

Phase 흐름 제어, 대상 선택, NEEDS_WORK 반영, endpoints.json 상태 관리. 서브에이전트가 무거운 작업을 하고, 메인은 조율만 한다.

4 프로젝트에 생기는 파일 

your-project/
├── PROJECT_CONTEXT.md                 ← Phase 0. 아키텍처/도메인 문서 (200줄)
├── docker-compose.test.yml            ← Phase 0. 테스트 전용 인프라 (볼륨 없음)
├── .test-coverage/
│   └── endpoints.json                 ← Phase 1. 진행 상태 추적 (git 커밋)
├── src/main/.../Controller.java       ← [AI-CONTEXT] 주석 추가됨
├── src/test/resources/
│   └── application-integration.yml    ← Phase 0. 테스트 프로필 설정
└── src/test/.../integration/          ← AI 전용 테스트 트랙
    ├── support/                       ← 베이스 클래스 + 공통 헬퍼
    ├── AuthControllerIT.java
    └── MemberControllerIT.java

5 설치 & 사용

설치 (1회)

git clone https://github.com/yyanoos/ai-workflow.git
cd ai-workflow
bash install.sh      # Mac/Linux
./install.ps1        # Windows

소스 수정 후 install 재실행하면 반영됩니다.

사용

cd your-project
/gen-api-tests

# 1회차 → Phase 0 (분석) → 멈춤
# 2회차 → Phase 1 (스캔) → 멈춤
# 3회차 → 엔드포인트 1개 생성+실행+리뷰 → 멈춤
# 4회차 → 승인 + 다음 엔드포인트 → 멈춤
# ...

플래그

POST /api/auth/login 특정 엔드포인트 지정
--reanalyze Phase 0 강제 재실행
--rescan Phase 1 강제 재실행
--retry-failed 실패 엔드포인트를 pending으로 리셋

6 설계 원칙 (gen-api-tests)

사람의 시간이 가장 비싸다

사람은 "통과하고 리뷰도 끝난 테스트"만 검수한다. 컴파일 에러, 실행 실패는 AI가 먼저 해결한다.

메인 세션은 오케스트레이터

무거운 작업은 서브에이전트가 처리. 메인 세션의 컨텍스트를 보호하고, 각 에이전트는 독립적으로 실행된다.

1개씩, 멈추고, 검수

한 번에 모든 테스트를 만들지 않는다. 엔드포인트 1개씩 처리하고 사람이 확인한 뒤 다음으로 넘어간다.

/dev — TDD 기반 개발

테스트 환경 구축 후, 기능 개발/수정/버그 수정을 TDD 방식으로 수행한다. 복수 세션 동시 가능.

1 한눈에 보는 흐름

Phase 1
구현 명세 메인 세션 대화형

개발자와 대화 → 변경 유형, 영향 범위, 스키마 변경, 수용 기준 합의

산출물: .dev/{slug}/spec.md + feature/{slug} 브랜치

멈춤
Phase 2
테스트 작성 spec-test-writer RED

명세 기반 통합 테스트 작성 → 컴파일 성공 + 테스트 실패 (구현 부재)

스키마 변경 시 마이그레이션 먼저 생성. 단위 테스트 X.

멈춤
Phase 3
구현 implementer GREEN

테스트를 통과시키는 코드 작성 → 전체 테스트 통과 (새 것 + 기존 것)

제약: 테스트 수정 금지, 기존 코드 리팩토링 금지, 막히면 중단 보고

 
Phase 4
코드 리뷰 code-reviewer

보안, 에러 처리, 성능, DB 안전성, 설계, 유지보수성 검토

NEEDS_WORK: minor → 메인 반영, major → implementer 재호출 (리뷰 지적사항이 지시문)

멈춤
Phase 5
MR + 알림 메인 세션

commit → push → MR 생성 (spec.md 기반 설명) → Slack 알림 (설정 시)

 

2 상태 머신

spec → red → green → reviewed → merged
구현 불가 → blocked → 사람 판단 → spec 또는 green
spec

명세 완료

red

테스트 실패 중

green

전체 통과

reviewed

리뷰 완료

blocked

구현 불가

merged

MR 완료

3 에이전트 구성

spec-test-writer Phase 2

명세 기반 통합 테스트 작성. 스키마 변경 시 마이그레이션도 생성. RED 확인까지.

Tools: Read, Write, Edit, Grep, Glob, Bash

implementer Phase 3

테스트를 통과시키는 코드 작성. 테스트 수정 금지. 막히면 중단 보고.

Tools: Read, Write, Edit, Grep, Glob, Bash

code-reviewer Phase 4

빅테크 시니어 관점 리뷰. 보안/성능/설계 검토. 읽기 전용.

Tools: Read, Grep, Glob

메인 세션 (오케스트레이터)

Phase 1 대화형 명세 작성, Phase 흐름 제어, NEEDS_WORK minor 반영, MR 생성. 서브에이전트가 무거운 작업을 하고, 메인은 조율만 한다.

4 프로젝트에 생기는 파일 

your-project/
├── .dev/
│   ├── board-crud/                    ← 기능별 독립 디렉토리 (동시 개발 가능)
│   │   ├── spec.md                    ← Phase 1. 구현 명세
│   │   └── status.json                ← 진행 상태 + feature 브랜치명
│   └── notification/                  ← 다른 세션에서 동시 작업 가능
│       ├── spec.md
│       └── status.json
├── src/main/.../BoardController.java  ← Phase 3. 새 프로덕션 코드
├── db/migration/V3__add_board.sql     ← Phase 2. 마이그레이션 (필요 시)
└── src/test/.../integration/
    └── BoardControllerIT.java         ← Phase 2. 명세 기반 테스트

5 사용법 

# 세션 A
/dev "게시판 CRUD API 추가"
# → .dev/board-crud/ + feature/board-crud 브랜치

# 세션 B (동시 가능)
/dev "알림 기능 추가"
# → .dev/notification/ + feature/notification 브랜치

# 이어서 하기 (기능명으로 자동 매칭)

# → .dev/board-crud/status.json의 상태에 따라 진입

플래그

--respec Phase 1 강제 재실행
--retest Phase 2 강제 재실행
--from 3 특정 Phase부터 시작

• 설계 원칙 (공통)

사람의 시간이 가장 비싸다

사람은 의도와 정확성만 검수한다. 컴파일, 실행, 인프라 검증은 AI가 먼저 해결한다.

메인 세션은 오케스트레이터

무거운 작업은 서브에이전트가 처리. 메인 세션의 컨텍스트를 보호하고, 각 에이전트는 독립적으로 실행된다.

테스트가 진실의 원천

테스트를 먼저 쓰고, 테스트를 통과시키는 구현을 하고, 테스트가 깨지지 않는 리뷰를 한다.
